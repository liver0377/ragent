# Ragent Python 版 — 通用基础设施层 (framework)

## 1. 模块定位

framework 是整个系统的最底层模块，不依赖任何业务模块，提供与业务无关的通用能力。所有上层模块（infra_ai、rag、ingestion、app）都依赖它。

## 3. Redis 封装

### 3.1 异步连接池

基于 redis-py 的异步模式，通过 FastAPI 生命周期管理连接池的创建和销毁。应用启动时建立连接池，关闭时自动释放。

```
┌───────────────────────────────────────┐
│          Redis Manager                │
│                                       │
│  ┌─────────────────────────────────┐  │
│  │  连接池 (ConnectionPool)         │  │
│  │  ├── 最大连接数可配置            │  │
│  │  ├── 超时时间可配置              │  │
│  │  └── 连接复用                    │  │
│  └─────────────────────────────────┘  │
│                                       │
│  ┌─────────────────────────────────┐  │
│  │  命名空间隔离                    │  │
│  │  ├── Key 格式: {prefix}:{name}  │  │
│  │  ├── 不同业务使用不同前缀        │  │
│  │  └── 避免键名冲突                │  │
│  └─────────────────────────────────┘  │
│                                       │
│  ┌─────────────────────────────────┐  │
│  │  常用操作封装                    │  │
│  │  ├── get / set / delete          │  │
│  │  ├── zadd / zrange (ZSET)       │  │
│  │  ├── hset / hget (HASH)         │  │
│  │  └── Lua 脚本原子操作            │  │
│  └─────────────────────────────────┘  │
└───────────────────────────────────────┘
```

### 3.2 分布式锁

基于 Redis 的分布式锁实现，用于保护临界资源（如并发入库时的冲突检测）：

```
获取锁
  │
  ▼
┌──────────────────────────┐
│ SET key value NX PX ttl  │
│ (原子操作，设置过期时间)   │
└─────────────┬────────────┘
              │
       ┌──────┴──────┐
       ▼             ▼
    获取成功       获取失败
       │             │
       ▼             ▼
  执行临界逻辑   重试或返回失败
       │
       ▼
  释放锁 (Lua 原子释放)
  ┌──────────────────────────┐
  │ 校验 value 一致性后删除    │
  │ 避免误删其他持有者的锁     │
  └──────────────────────────┘
```

## 4. MQ 封装

### 4.1 架构概览

基于 RabbitMQ + aio-pika 实现异步消息队列，封装生产者和消费者的通用逻辑。

```
┌──────────────────────────────────────────────────────────┐
│                    MQ 封装层                               │
│                                                           │
│  ┌─────────────────────────────────────────────────────┐ │
│  │  Producer (生产者)                                    │ │
│  │  ├── 连接管理（自动重连）                              │ │
│  │  ├── 消息序列化（JSON）                               │ │
│  │  ├── 消息包装（统一信封格式）                          │ │
│  │  └── 发布确认（publisher confirms）                   │ │
│  └─────────────────────────────────────────────────────┘ │
│                                                           │
│  ┌─────────────────────────────────────────────────────┐ │
│  │  Consumer (消费者)                                    │ │
│  │  ├── 连接管理（自动重连）                              │ │
│  │  ├── 预取控制（QoS）                                  │ │
│  │  ├── 消息反序列化                                     │ │
│  │  ├── 幂等消费（Redis 去重）                           │ │
│  │  ├── 死信队列（消费失败转入 DLX）                     │ │
│  │  └── 手动 ACK / NACK                                 │ │
│  └─────────────────────────────────────────────────────┘ │
│                                                           │
│  ┌─────────────────────────────────────────────────────┐ │
│  │  统一消息信封 (MessageWrapper)                         │ │
│  │  ├── message_id（消息唯一ID，用于幂等）                │ │
│  │  ├── message_type（消息类型标识）                      │ │
│  │  ├── timestamp（发送时间戳）                           │ │
│  │  ├── trace_id（链路追踪ID）                            │ │
│  │  └── payload（业务数据）                               │ │
│  └─────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────┘
```

### 4.2 消息流转流程

```
业务层发送消息
       │
       ▼
┌──────────────┐
│  Producer    │
│  封装消息信封 │
│  序列化      │
│  发布到Exchange│
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  RabbitMQ    │
│  Exchange    │──── 路由到 Queue
│  (Topic模式) │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  Consumer    │
│  反序列化    │
│  幂等检查    │──── Redis 去重
│  执行业务    │
│  ACK/NACK    │
└──────────────┘
       │ 失败
       ▼
┌──────────────┐
│  Dead Letter │
│  Exchange    │
│  (DLX)       │
└──────────────┘
```

### 4.3 业务使用的队列

| 队列名称                  | 用途                     | 生产者           | 消费者          |
|---------------------------|--------------------------|------------------|-----------------|
| ingestion.task            | 文档入库任务分发          | 文档上传接口      | 入库 Pipeline   |
| ingestion.chunk           | 分块结果异步处理          | Chunker 节点      | Indexer 节点    |
| rag.feedback              | 用户反馈异步记录          | 聊天接口          | 反馈处理器      |

### 4.4 幂等消费机制

```
消息到达
   │
   ▼
┌────────────────────────────┐
│ 提取 message_id            │
└─────────────┬──────────────┘
              │
              ▼
┌────────────────────────────┐
│ Redis SETNX 检查           │
│ key = idempotent:{msg_id}  │
│ ttl = 消费超时时间 × 2      │
└──────┬──────────┬──────────┘
       │          │
    设置成功    已存在
       │          │
       ▼          ▼
  执行业务    跳过（重复消费）
       │
       ▼
  ACK 确认
```

## 5. 异常体系

三级异常，对应不同错误来源：

```
┌──────────────────────────────────────────────────────┐
│                    BaseError                          │
│                (错误码 + 消息)                         │
├──────────────────────────────────────────────────────┤
│                                                      │
│  ┌──────────────┐  ┌───────────────┐  ┌────────────┐│
│  │ClientException│  │ServiceException│ │RemoteException│
│  │ 客户端错误    │  │ 服务端错误     │  │ 远程调用错误 ││
│  │ 4xx          │  │ 5xx           │  │ 外部服务故障 ││
│  │ 参数校验失败  │  │ 业务逻辑异常  │  │ 模型调用失败 ││
│  │ 权限不足      │  │ 数据不一致    │  │ 网络超时     ││
│  └──────────────┘  └───────────────┘  └────────────┘│
└──────────────────────────────────────────────────────┘
```

错误码规范：

| 错误码前缀 | 含义         | 示例                     |
|-----------|-------------|--------------------------|
| A         | 客户端错误   | A1001: 参数校验失败       |
| B         | 服务端错误   | B2001: 知识库不存在       |
| C         | 远程调用错误 | C3001: 模型服务不可用     |

## 6. 分布式 ID

采用 Snowflake 算法，通过 Redis Lua 脚本在应用启动时原子分配 worker_id：

```
应用启动
   │
   ▼
┌──────────────────────────┐
│ Redis Lua 脚本            │
│ 原子分配 worker_id        │
│ (自增计数，防止冲突)       │
└─────────────┬────────────┘
              │
              ▼
┌──────────────────────────┐
│ Snowflake ID 生成器       │
│                           │
│ ┌───┬───┬───────┬──────┐ │
│ │ 1 │41 │ 10    │ 12   │ │
│ │符号│时间│worker│序列号│ │
│ │位  │戳  │_id   │      │ │
│ └───┴───┴───────┴──────┘ │
│                           │
│ 趋势递增 + 全局唯一        │
└──────────────────────────┘
```

## 7. 用户上下文

基于 Python ContextVar 实现协程安全的用户上下文透传，确保在异步场景下用户身份不丢失：

```
请求进入（中间件）
   │
   ▼
┌──────────────────────────┐
│ 解析 Token               │
│ 设置 ContextVar           │
│ user_id / username / role │
└─────────────┬────────────┘
              │
              ▼
        业务层任意位置
        可直接读取上下文
        无需显式传参
              │
              ▼
┌──────────────────────────┐
│ async with 方式           │
│ 自动清理上下文             │
│ 支持嵌套覆盖               │
└──────────────────────────┘
```

## 8. 链路追踪

注解驱动的链路追踪框架，覆盖 RAG 问答的每一个环节：

```
@rag_trace_root("rag-chat")
  ├── @rag_trace_node("query-rewrite")     — 问题重写耗时
  ├── @rag_trace_node("intent-classify")   — 意图分类耗时
  ├── @rag_trace_node("retrieval")         — 多路检索耗时
  │     ├── @rag_trace_node("channel-1")   — 单通道耗时
  │     └── @rag_trace_node("channel-2")   — 单通道耗时
  ├── @rag_trace_node("rerank")            — 重排序耗时
  ├── @rag_trace_node("prompt-build")      — Prompt 组装耗时
  └── @rag_trace_node("llm-generate")      — LLM 生成耗时
```

追踪数据通过 ContextVar 在协程间透传 trace_id，所有子节点自动关联到同一个 trace_run 记录。
