# Ragent Python 版 — 通用基础设施层 (framework)

## 1. 模块定位

framework 是整个系统的最底层模块，不依赖任何业务模块，提供与业务无关的通用能力。所有上层模块（infra_ai、rag、ingestion、app）都依赖它。

## 10. JSON 序列化安全（BigInt 保护）

### 10.1 问题背景

JavaScript `Number` 类型的安全整数范围为 -(2^53 - 1) ~ 2^53 - 1（即 ±9007199254740991）。本项目使用 Snowflake 算法生成分布式 ID，ID 长度为 18-19 位十进制数字，超出 JS 安全范围。如果 JSON 响应中直接返回大整数，前端 `JSON.parse` 后会发生精度丢失。

### 10.2 LargeIntJSONEncoder

位于 `ragent/common/json_utils.py`，自定义 JSON 编码器，在序列化前递归遍历数据结构，将超出 JS 安全范围的大整数自动转为字符串：

```
原始数据
   │
   ▼
┌──────────────────────────────────────┐
│ _convert_large_ints(obj)            │
│                                      │
│  isinstance(obj, int)                │
│    └── abs(obj) > 2^53 - 1 ?        │
│         ├── 是 → 转为 str(obj)       │
│         └── 否 → 保留原值            │
│                                      │
│  isinstance(obj, dict)               │
│    └── 递归处理每个 value            │
│                                      │
│  isinstance(obj, list / tuple)       │
│    └── 递归处理每个 element          │
│                                      │
│  其他类型 → 原样返回                  │
└──────────────────────────────────────┘
   │
   ▼
LargeIntJSONEncoder.encode() / iterencode()
   │
   ▼
安全 JSON 字符串（大整数已为字符串）
```

**核心常量：** `_JS_MAX_SAFE_INT = 9007199254740991`（即 2^53 - 1）

### 10.3 SafeJSONResponse

位于 `ragent/common/safe_json.py`，继承自 Starlette `JSONResponse`，使用 `LargeIntJSONEncoder` 渲染响应体：

- **覆盖 `render()` 方法：** 调用 `json.dumps(content, cls=LargeIntJSONEncoder, ensure_ascii=False, allow_nan=False)` 后编码为 UTF-8 字节
- **全局生效：** 在 `create_app()` 中通过 `default_response_class=SafeJSONResponse` 设为默认响应类，所有 API 端点自动获得 BigInt 保护
- **前端约定：** 前端接收到字符串类型的 ID 后以字符串形式传递，避免任何精度问题

## 11. 中间件体系

### 11.1 注册顺序与执行链

FastAPI 中间件采用"洋葱模型"——**最后添加的中间件最先执行**。在 `create_app()` 中的注册代码及实际执行顺序如下：

```
注册代码（从上到下）           执行顺序（请求进入方向）
─────────────────            ──────────────────────
① add_middleware(CORSMiddleware)     ← 最外层
② add_middleware(RateLimitMiddleware)
③ add_middleware(ExceptionHandlerMiddleware)
④ add_middleware(RequestContextMiddleware)
⑤ add_middleware(TraceMiddleware)    ← 最内层（最先与路由交互）
                                    → 路由 Handler
```

完整的请求处理流程：

```
请求进入
   │
   ▼
┌──────────────────────────────────┐
│ ① CORSMiddleware                 │
│   处理跨域预检 / 注入 CORS 头    │
└──────────────┬───────────────────┘
               │
               ▼
┌──────────────────────────────────┐
│ ② RateLimitMiddleware            │
│   IP + 路径前缀 滑动窗口限流     │
│   超限返回 429                   │
└──────────────┬───────────────────┘
               │
               ▼
┌──────────────────────────────────┐
│ ③ ExceptionHandlerMiddleware     │
│   捕获未处理异常                 │
│   转为 Result.error() 统一格式   │
└──────────────┬───────────────────┘
               │
               ▼
┌──────────────────────────────────┐
│ ④ RequestContextMiddleware       │
│   提取 X-User-Id / X-Username   │
│   / X-User-Role → ContextVar    │
└──────────────┬───────────────────┘
               │
               ▼
┌──────────────────────────────────┐
│ ⑤ TraceMiddleware                │
│   生成 / 传播 trace_id          │
│   OTel Span 记录                 │
│   写入响应头 X-Trace-Id         │
└──────────────┬───────────────────┘
               │
               ▼
         路由 Handler
```

### 11.2 各中间件职责

**CORSMiddleware**（FastAPI 内置）

- 通过 `settings.CORS_ORIGINS` 配置允许的源（逗号分隔）
- 允许所有方法和头部，携带凭证
- 放在最外层确保预检请求优先处理

**RateLimitMiddleware**

- 基于 Redis 滑动窗口（Sorted Set），按 IP + 路径前缀分组限流
- 保护关键端点：注册（5次/分）、登录（10次/分）、聊天（20次/分）、上传（10次/分）
- Redis Key 格式：`ratelimit:{ip}:{prefix}`，值为 Sorted Set（timestamp → request_id）
- Redis 不可用时跳过限流检查，不阻塞请求
- 超限返回 HTTP 429 + `Retry-After` 响应头

**ExceptionHandlerMiddleware**

- 捕获路由处理中未被显式捕获的异常
- 转换为 `Result.error(code=500, message="内部服务错误")` 格式的 JSON 响应
- 记录错误日志，附带 trace_id 和请求路径

**RequestContextMiddleware**

- 从请求头提取 `X-User-Id`、`X-Username`、`X-User-Role`
- 构造 `UserContext` 并写入 `ContextVar`，后续业务层可直接读取
- 请求结束后自动清理上下文

**TraceMiddleware**

- 为每个请求生成或传播 `trace_id`
- 若请求头携带 `X-Trace-Id` 则复用，支持分布式追踪传播
- 基于 OpenTelemetry Span 记录 HTTP 方法、URL、状态码
- trace_id 写入 ContextVar 供日志过滤器自动注入
- 响应头中返回 `X-Trace-Id`

**JWT 配置项说明：**

- **JWT_SECRET_KEY** — Token 签名密钥，默认值仅供开发使用，生产环境务必通过环境变量更换
- **JWT_ALGORITHM** — 签名算法，默认 HS256（对称加密），适合单体应用
- **JWT_ACCESS_TOKEN_EXPIRE_MINUTES** — Access Token 有效时长，默认 1440 分钟（24 小时）

**限流配置项说明：**

- **RATE_LIMIT_MAX_CONCURRENT** — 单 IP 最大并发请求数，默认 10
- **RATE_LIMIT_WINDOW_SECONDS** — 滑动窗口时长（秒），默认 60
