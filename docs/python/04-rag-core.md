# Ragent Python 版 — RAG 问答核心 (rag)

## 1. 模块定位

rag 模块是 Ragent 的核心业务模块，负责将用户提问经过问题重写、意图识别、多路检索、Prompt 组装、模型生成等步骤，最终通过 SSE 流式返回回答。

## 3. 问题重写

### 3.1 重写流程

```
原始用户提问
     │
     ▼
┌──────────────────────────────┐
│ 提取对话历史                  │
│ (近 N 轮，从会话记忆中获取)   │
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│ 上下文补全                    │
│ LLM 根据历史补全指代词和省略  │
│ 例: "它怎么用" → "XX产品怎么用"│
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│ 关键词归一化                  │
│ 查询 t_query_term_mapping 表 │
│ 将同义词映射到标准术语        │
│ 例: "AI" → "人工智能"        │
└──────────────┬───────────────┘
               │
               ▼
┌──────────────────────────────┐
│ 复杂问题拆分（可选）          │
│ 如果问题包含多个子问题        │
│ 拆分为独立子问题并行处理      │
└──────────────┬───────────────┘
               │
               ▼
         重写后的问题
```

## 4. 意图识别体系

### 4.1 树形意图结构

```
Level 0: ROOT（根）
    ├── Level 1: DOMAIN（领域）
    │       ├── Level 2: TOPIC（话题/叶节点）
    │       ├── Level 2: TOPIC（话题/叶节点）
    │       └── ...
    ├── Level 1: DOMAIN（领域）
    │       └── ...
    └── ...
```

每个叶节点（TOPIC）通过 `kind` 字段区分类型：
- `kind=0`（RAG）→ 关联 `collection_name`，走 RAG 检索路径
- `kind=1`（TOOL）→ 走工具调用路径（预留，尚未实现）

### 4.2 意图分类流程

```
                        用户问题
                           │
                           ▼
                  ┌─────────────────┐
                  │ IntentClassifier │
                  │ (LLM 打分)       │
                  └────────┬────────┘
                           │
                           ▼
              对所有叶节点打分 + 置信度排序
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
         高置信度       中置信度       低置信度
         (≥ 阈值)     (歧义区间)     (< 阈值)
              │            │            │
              ▼            ▼            ▼
        直接执行       歧义检测       全局检索
        指定意图       与引导        (不限意图)
              │            │
              │     ┌──────┴──────┐
              │     ▼             ▼
              │  引导用户澄清   置信度 OK
              │  (返回选项)    (执行意图)
              │                  │
              └──────────────────┘
                        │
                        ▼
                  确定执行路径
                  └── RAG 检索 → 指定 Collection
                      （工具调用路径预留，尚未实现）
```

### 4.3 LLM 打分机制

```
┌──────────────────────────────────────────────────────┐
│  LLM 意图打分流程                                      │
│                                                       │
│  1. 收集所有叶节点（TOPIC）的名称和描述                 │
│  2. 将用户问题 + 叶节点列表发给 LLM                    │
│  3. LLM 为每个叶节点输出相关性分数 (0-1)               │
│  4. 按分数降序排列                                     │
│  5. 取 Top-1 分数:                                    │
│     ├── ≥ 高阈值 (如 0.8) → 高置信度，直接采用        │
│     ├── ≥ 低阈值 (如 0.5) → 中置信度，歧义检测        │
│     └── < 低阈值       → 低置信度，走全局检索          │
│                                                       │
│  歧义检测:                                            │
│  Top-1 与 Top-2 分数差距 < 距离阈值                    │
│  → 认为存在歧义，引导用户从候选中选择                  │
└──────────────────────────────────────────────────────┘
```

## 5. 多路检索引擎

### 5.1 并行检索架构

基于 asyncio.gather 实现多通道并行检索：

```
                         重写后的问题
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
     ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
     │ 意图定向检索  │ │ 全局向量检索  │ │  (可扩展...)  │
     │ 通道          │ │ 通道          │ │              │
     │              │ │              │ │              │
     │ 按意图路由到  │ │ 全量 Collection│ │              │
     │ 指定Collection│ │ 并行检索      │ │              │
     └──────┬───────┘ └──────┬───────┘ └──────┬───────┘
            │                │                │
            └────────────────┼────────────────┘
                             ▼
                    ┌─────────────────┐
                    │   结果合并       │
                    └────────┬────────┘
                             ▼
                    ┌─────────────────┐
                    │  去重后处理器     │
                    │ (跨通道去重)     │
                    └────────┬────────┘
                             ▼
                    ┌─────────────────┐
                    │  重排序后处理器   │
                    │ (Rerank模型)    │
                    └────────┬────────┘
                             ▼
                       最终检索结果
```

### 5.2 检索通道

| 通道           | 说明                                  | 调度方式     |
|---------------|---------------------------------------|-------------|
| 意图定向检索   | 根据意图分类结果，只检索关联的 Collection | 条件触发    |
| 全局向量检索   | 在全量 Collection 中做向量近邻搜索      | 始终执行    |
| 可扩展通道     | 实现新的 SearchChannel 接口即可接入     | 灵活配置    |

### 5.3 后处理器链

```
┌──────────────────────────────────────────────────────┐
│  后处理器链（串联执行）                                │
│                                                       │
│  Step 1: 去重后处理器 (DeduplicatePostProcessor)       │
│  ├── 按 content_hash 跨通道去重                       │
│  ├── 保留分数最高的副本                               │
│  └── 输入: 合并结果列表 → 输出: 去重后列表             │
│                                                       │
│  Step 2: 重排序后处理器 (RerankPostProcessor)          │
│  ├── 调用 llm_service 对结果重新评分（当前为 Mock）    │
│  ├── 按相关性分数重新排序                              │
│  └── 输入: 去重后列表 → 输出: 精排后列表               │
│                                                       │
│  注: 当前实现为固定两步流水线（去重 → 重排序）         │
│  后处理器无公共基类，扩展时需同时修改 RetrievalEngine   │
└──────────────────────────────────────────────────────┘
```

## 6. 会话记忆管理

### 6.1 滑动窗口 + 自动摘要

```
                    当前对话
                       │
                       ▼
              ┌─────────────────┐
              │ 滑动窗口         │
              │ (保留近 N 轮)    │
              └────────┬────────┘
                       │
              ┌────────┴────────┐
              │  轮数是否超限？   │
              └────────┬────────┘
                  │           │
              未超限         超限
                  │           │
                  ▼           ▼
           直接使用      ┌──────────────┐
           历史          │ 自动摘要压缩  │
                        │ (LLM 生成)   │
                        │ 摘要持久化到  │
                        │ summary 表   │
                        └──────┬───────┘
                               │
                               ▼
                        摘要 + 最近 N 轮
                        组装为上下文
```

### 6.2 记忆存储结构

```
┌──────────────────────────────────────────────────────┐
│  会话记忆层次                                         │
│                                                       │
│  Level 1: 短期记忆（当前为内存存储，后续接入数据库）   │
│  ├── 最近 N 条完整对话（默认 20 条 ≈ 10 轮）          │
│  └── N 值可配置                                      │
│                                                       │
│  Level 2: 长期记忆（摘要压缩）                        │
│  ├── 超出窗口的历史对话被压缩为摘要                    │
│  ├── 当前摘要存储在内存中                             │
│  ├── 后续将持久化到 t_conversation_summary 表         │
│  └── LLM 生成摘要，保留关键信息                       │
│                                                       │
│  上下文组装:                                          │
│  [长期摘要] + [最近N轮对话] + [当前问题]              │
│  → 发送给 LLM                                        │
└──────────────────────────────────────────────────────┘
```

## 8. 并行调度策略

使用 asyncio 替代 Java 的线程池实现并行调度：

| 场景             | Java 版                    | Python 版                  |
|-----------------|----------------------------|----------------------------|
| 多路检索并行     | ThreadPoolExecutor         | asyncio.gather             |
| 单通道内并行     | 内部线程池                 | asyncio.gather             |
| 意图分类         | 意图分类线程池             | 协程直调                   |
| 记忆摘要         | 记忆摘要线程池             | 后台协程 (asyncio.create_task) |
| 模型流式输出     | OkHttp 回调线程            | 异步生成器 (async for)     |

## 9. 扩展点设计

| 扩展能力       | 方式                                | 说明                         |
|---------------|-------------------------------------|------------------------------|
| 新增检索通道   | 实现 SearchChannel 接口             | 自动参与多路检索              |
| 新增后处理器   | 创建独立后处理类并注册到 RetrievalEngine | 当前无公共基类，需手动集成 |
| 自定义重写策略 | 继承 QueryRewriter 类并重写 rewrite 方法 | 替换默认重写逻辑              |
| 自定义意图树   | 数据库配置意图节点                   | 运行时修改，无需重启          |
| 新增 Prompt 模板 | 按意图节点配置不同模板             | 不同意图走不同 Prompt 策略    |
