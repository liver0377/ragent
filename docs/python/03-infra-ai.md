# Ragent Python 版 — AI 模型基础设施层 (infra_ai)

## 1. 模块定位

infra_ai 模块对上层业务提供统一的 AI 能力接口，对下层屏蔽不同模型供应商的协议差异。基于 litellm 实现多供应商统一调用，配合 circuitbreaker 实现模型级熔断降级。

## 4. 模型路由机制

模型路由确保系统不依赖单一供应商，核心流程：

```
用户请求
    │
    ▼
ModelSelector.select_candidates()
    │
    ├── 读取配置的候选模型列表
    ├── 按优先级排序
    ├── 过滤掉熔断状态为 OPEN 的模型
    │
    ▼
RoutingExecutor.execute_with_fallback()
    │
    ├── 候选 1 → litellm 调用 → 成功？ → 返回结果
    │                          失败？ → 记录失败，尝试下一个
    ├── 候选 2 → litellm 调用 → 成功？ → 返回结果
    │                          失败？ → 记录失败，尝试下一个
    ├── 候选 3 → litellm 调用 → ...
    │
    ▼
全部失败 → 抛出 RemoteException
```

## 5. 三态熔断器

基于 circuitbreaker 库实现模型级别的熔断保护：

```
         失败次数 < 阈值                失败次数 ≥ 阈值
    ┌──────────┐  (正常调用)   ┌──────────┐  (触发熔断)
    │  CLOSED  │──────────────▶│   OPEN   │
    │ (允许调用) │              │ (拒绝调用) │
    └──────────┘               └────┬─────┘
         ▲                          │
         │    探测成功               │ 冷却期结束
         │                          ▼
         │                    ┌───────────┐
         └────────────────────│ HALF_OPEN │
              探测成功         │ (放行1个   │
              恢复 CLOSED      │  探测请求) │
                               └───────────┘
                                    │ 探测失败
                                    ▼
                              回到 OPEN 状态
```

熔断器参数：

| 参数             | 说明                           | 默认值   |
|-----------------|-------------------------------|----------|
| failure_threshold| 触发熔断的连续失败次数          | 5        |
| recovery_timeout | OPEN → HALF_OPEN 的冷却时间    | 60s      |
| success_threshold| HALF_OPEN → CLOSED 的成功次数   | 3        |

每个模型供应商维护独立的熔断器实例，互不影响。

## 6. 流式首包探测

流式对话场景下的首包探测，确保模型切换时用户不会收到半截数据：

```
RoutingLLMService.stream_chat()
    │
    ├── 候选 1 → ProbeStreamBridge 包装
    │            │
    │            ├── 等待首包（最长 60s）
    │            ├── 首包到达 → 刷新缓冲区，后续直接透传 → 返回
    │            ├── 首包超时/错误 → 取消流，标记失败 → 尝试下一个
    │
    ├── 候选 2 → ProbeStreamBridge 包装
    │            └─ ...（同上）
    │
    ▼
全部失败 → 通过 SSE 推送 error 事件
```

ProbeStreamBridge 是一个异步生成器装饰器，工作流程：

```
┌──────────────────────────────────────────────────────────┐
│  ProbeStreamBridge 工作流程                                │
│                                                           │
│  1. 创建内部缓冲队列                                       │
│  2. 启动 litellm 流式调用（后台协程）                      │
│  3. 等待第一个 token（带超时）                              │
│     ├── 首包到达                                           │
│     │   ├── 将首包内容 yield 给上层                       │
│     │   └── 后续 token 直接 yield（透传模式）              │
│     ├── 首包超时                                           │
│     │   ├── 取消后台协程                                   │
│     │   ├── 标记该候选失败                                 │
│     │   └── RoutingExecutor 尝试下一个候选                 │
│     └── 首包错误（API 异常等）                             │
│         └── 同超时处理流程                                 │
└──────────────────────────────────────────────────────────┘
```

## 8. 扩展点设计

| 扩展能力           | 方式                                  | 说明                         |
|--------------------|---------------------------------------|------------------------------|
| 新增模型供应商     | litellm 已内置 100+ 供应商支持        | 只需配置 model_name 即可      |
| 自定义路由策略     | 实现 ModelSelector 接口               | 替换默认优先级排序逻辑        |
| 自定义熔断策略     | 配置 circuitbreaker 参数              | 按模型级别独立配置            |
| 新增 Rerank 供应商 | litellm.rerank 支持                   | 添加到候选列表即可            |
| 流式回调定制       | 实现 StreamCallback 接口              | 自定义 on_content / on_error 等 |
