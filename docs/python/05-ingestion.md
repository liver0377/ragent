# Ragent Python 版 — 文档入库流水线 (ingestion)

## 1. 模块定位

ingestion 模块负责将用户上传的文档经过解析、增强、分块、丰富、向量化的完整流水线，最终写入向量数据库，使其可被检索。

**核心设计原则：所有入库操作均通过 MQ 异步执行，不提供同步入库路线。**

---

                                                            │
                                           ┌────────────────┘

              ┌────────────────────────────┼────────────────────────────┐
              ▼                            ▼                            ▼

## 3. MQ 异步入库流程

### 3.1 任务提交阶段

```
用户上传文件
     │
     ▼
┌──────────────────────────────────────┐
│  API 接口                             │
│  ├── 校验文件类型与大小                │
│  ├── 生成 task_id（Snowflake）        │
│  ├── 写入数据库任务记录（状态: PENDING）│
│  ├── 将文件存储到临时目录              │
│  └── 发送消息到 ingestion.task 队列    │
│      消息体: task_id + 文件路径 + 配置  │
└──────────────────────────────────────┘
     │
     ▼
  立即返回 task_id 给用户
  （用户可轮询/订阅任务状态）
```

### 3.2 Pipeline 执行阶段

消费者收到消息后，启动 Pipeline 引擎：

```
┌──────────────────────────────────────────────────────────────┐
│                    Pipeline 引擎                               │
│                                                               │
│  1. 从消息中提取 task_id 和 Pipeline 配置                      │
│  2. 构建 NodeConfig 链表（通过 next_node_id 串联）             │
│  3. 执行环路检测                                               │
│  4. 找到起始节点，依次执行:                                     │
│     ├── ConditionEvaluator 评估条件                            │
│     ├── 跳过条件不满足的节点                                    │
│     ├── 调用对应节点的 execute() 方法                           │
│     ├── 将输出写入 IngestionContext                             │
│     └── 记录节点执行日志到数据库                                │
│  5. 更新任务状态为 COMPLETED 或 FAILED                         │
└──────────────────────────────────────────────────────────────┘
```

### 3.3 消息确认与容错

```
消费者处理消息
       │
       ├── 成功 → ACK 确认
       │
       ├── 业务异常 → 记录失败日志 → ACK（不重试，避免死循环）
       │
       └── 系统异常 → NACK + requeue → 重试
              │
              ├── 重试次数 < 阈值 → 重新入队
              └── 重试次数 ≥ 阈值 → 转入死信队列 (DLX)
                                        │
                                        ▼
                                   人工介入处理
```

## 4. Pipeline 节点详解

### 4.1 Fetcher — 数据抓取

从数据源获取原始文件字节：

```
┌─────────────────────────────────────┐
│          Fetcher 节点                │
│                                      │
│  策略模式，支持多种数据源:            │
│  ├── LocalFile  — 本地文件系统       │
│  ├── HttpUrl    — HTTP/HTTPS URL     │
│  └── S3Storage  — S3 兼容对象存储   │
│                                      │
│  输入: source_type + source_location │
│  输出: raw_bytes (原始文件字节流)     │
└─────────────────────────────────────┘
```

### 4.2 Parser — 文档解析

将原始文件解析为纯文本：

```
┌─────────────────────────────────────┐
│          Parser 节点                 │
│                                      │
│  根据 file_type 选择解析器:          │
│  ├── PDF      → pdfplumber          │
│  ├── DOCX     → python-docx         │
│  ├── Markdown → 直接读取            │
│  └── 其他     → unstructured 通用   │
│                                      │
│  输入: raw_bytes + file_type         │
│  输出: plain_text (纯文本)           │
└─────────────────────────────────────┘
```

### 4.3 Enhancer — 文档增强

通过 LLM 对原始文本进行增强处理：

```
┌─────────────────────────────────────┐
│          Enhancer 节点               │
│                                      │
│  LLM 增强（可选节点）:               │
│  ├── 上下文补充 — 补全缺失信息       │
│  ├── 关键词提取 — 提取核心关键词     │
│  └── 问题生成  — 生成 FAQ 问答对    │
│                                      │
│  输入: plain_text                    │
│  输出: enhanced_text + keywords      │
└─────────────────────────────────────┘
```

### 4.4 Chunker — 文本分块

将长文本按策略切分为小块，并调用 Embedding 模型向量化：

```
┌─────────────────────────────────────┐
│          Chunker 节点                │
│                                      │
│  分块策略:                           │
│  ├── FixedSize — 固定大小+可配置重叠  │
│  └── StructureAware — 按标题/章节切分│
│                                      │
│  向量化:                             │
│  每个分块调用 EmbeddingService       │
│  生成向量表示                        │
│                                      │
│  输入: enhanced_text + 分块策略配置   │
│  输出: chunks[] (含 content + vector)│
└─────────────────────────────────────┘
```

### 4.5 Enricher — 分块丰富

对每个分块进行二次 LLM 处理，提升检索质量：

```
┌─────────────────────────────────────┐
│          Enricher 节点               │
│                                      │
│  逐块处理（可选节点）:               │
│  ├── 提取块级关键词                  │
│  ├── 生成块级摘要                    │
│  └── 生成块级问题                    │
│                                      │
│  输入: chunks[]                      │
│  输出: enriched_chunks[]             │
└─────────────────────────────────────┘
```

### 4.6 Indexer — 向量入库

将分块写入向量数据库：

```
┌─────────────────────────────────────┐
│          Indexer 节点                │
│                                      │
│  双引擎支持:                         │
│  ├── Milvus  — 生产环境推荐         │
│  └── pgvector — 轻量部署备选        │
│                                      │
│  写入内容:                           │
│  ├── 向量 (embedding)               │
│  ├── 原文 (content)                 │
│  └── 元数据 (metadata: doc_id, 等)  │
│                                      │
│  输入: enriched_chunks[]             │
│  输出: 入库确认                      │
└─────────────────────────────────────┘
```

## 5. MQ 队列设计

### 5.1 Exchange 与 Queue 拓扑

```
┌───────────────────────────────────────────────────────────┐
│                    RabbitMQ 拓扑                           │
│                                                           │
│  ┌─────────────┐                                         │
│  │ ingestion   │                                         │
│  │ Exchange    │                                         │
│  │ (Topic)     │                                         │
│  └──┬────┬────┬┘                                         │
│     │    │    │                                           │
│     │    │    └────────────────────┐                     │
│     │    │                         │                     │
│     ▼    ▼                         ▼                     │
│  ┌──────┐┌──────┐          ┌──────────────┐              │
│  │task  ││chunk │          │ingestion.dlx │              │
│  │Queue ││Queue │          │(死信队列)     │              │
│  └──────┘└──────┘          └──────────────┘              │
│                                                           │
│  路由键:                                                  │
│  ingestion.task     → 入库任务分发                         │
│  ingestion.chunk    → 分块结果处理                         │
│  ingestion.dlx      → 失败消息转入死信                     │
└───────────────────────────────────────────────────────────┘
```

### 5.2 任务状态流转

```
PENDING ──▶ RUNNING ──┬──▶ COMPLETED
                      │
                      └──▶ FAILED ──▶ (可重试) ──▶ PENDING
```

| 状态       | 说明                       |
|-----------|----------------------------|
| PENDING   | 已提交，等待消费            |
| RUNNING   | Pipeline 执行中             |
| COMPLETED | 入库完成                    |
| FAILED    | 入库失败，记录错误信息       |

## 6. Pipeline 引擎扩展点

| 扩展能力       | 方式                           | 说明                       |
|---------------|-------------------------------|----------------------------|
| 新增入库节点   | 实现 IngestionNode 接口        | 可插入 Pipeline 任意位置    |
| 新增数据抓取   | 实现 Fetcher 接口             | 支持新的数据源类型          |
| 新增分块策略   | 实现 ChunkingStrategy 接口    | 支持新的分块算法            |
| 新增解析器     | 实现 Parser 接口              | 支持新的文件格式            |
| 条件节点       | 通过 condition_json 配置       | 根据条件决定是否执行        |

---

## 7. 设计原则

- **全异步**：所有入库操作通过 MQ 异步执行，API 接口立即返回
- **可追溯**：每个任务的每个节点都有独立的执行日志记录
- **可恢复**：失败任务可重试，死信队列兜底
- **可插拔**：Pipeline 节点通过配置组合，新增节点无需修改引擎
- **条件执行**：每个节点支持条件判断，按需跳过
